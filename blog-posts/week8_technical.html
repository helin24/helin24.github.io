Big O Notation

07/27/14

In mathematics, Big O notation describes how a function acts when approaching a specific value or when approaching infinity, by stating a simplified function whose growth rate will be just higher than the growth rate of the original function.

In computer science, Big O notation is used to describe the average and maximum amount of time needed for an algorithm to run. The time required is proportional to the number of basic operations performed - examples of basic operations are one arithmetic operation (e.g. 2+3), one variable assignment (e.g. x=3), one test (x == 1), one read, or one write. The concept of Big O notation can be understood through an example of a simple algorithm.

In a search algorithm, we may have an array of size n in which we are searching for a specific value. A simple solution is to iterate over the array's elements in order, checking each element to see if it is equal to the specific value. Depending on the array's contents, we could find the specific value in one basic operation (one test of equality) if the first element satisfies our condition, but it could also take up to n basic operations (n tests of equality) if none of the elements satisfies our condition. Typically, we are more interested in the worst case, so we can use Big O notation to describe the algorithm as having O(n) complexity - meaning that at worst, the number of tests is a multiple of n, the number of array elements.

Another solution is to use a binary search algorithm, which requires a sorted array. Instead of starting by checking the first element, though, the binary search algorithm compares the middle element to the target value. If the middle element is smaller(larger), the algorithm then checks the element halfway between the middle element and the end(beginning) of the array. This process is repeated until the target value is found or the algorithm finds that there are two adjacent elements that would sandwich the target value but the target value is not present. This algorithm is an improvement over the solution that iterates over all of the array elements, because the worst case number of basic operations (individual tests) is equal to log(n) (log base 2). For example, if you have an array of 64 elements, you can split it in half at most 6 times (2^6 is 64).
