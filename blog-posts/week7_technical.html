SQL vs. NoSQL

07/20/14

We've been learning about SQL databases in this week, so we can use this understanding as a base for contrasting NoSQL (stands for Not Only SQL).

SQL is "relational"
  • Table-based
  • Standard schema defined
  • Follows ACID
    • Atomicity - each transaction is success or failure only
    • Consistency - any transaction brings database from one valid state to another valid state
    • Isolation - transactions execute independently of each other
    • Durability - changes are permanent
  • Standard query language exists
  • Best used when transactions must be definite
  • Examples include MySQL, Oracle, SQLite
NoSQL is "distributed"
  • Key-value pairs, documents, etc.
  • No standard schema defined
  • Follows BASE
    • Basically Available - guarantees database is accessible at any time
    • Soft state - state may change over time even without input
    • Eventual consistency - system will become the same for everyone over time


  • No standard query language
  • Best used when many simple queries and changing structure
  • Examples include MongoDB, BigTable

The major difference between SQL and NoSQL databases is that SQL is highly structured into a table form, while NoSQL databases can take many forms. This is commonly a key-value structure where each piece of data is a value and a key is assigned to retrive the data later, or a document database where each document has a different collection of fields with information. Each SQL database table thus has a standard schema, where each field is defined to take a certain type of data (e.g. integers, varchar, date) and default values are specified. NoSQL database objects do not have defined schema, and each piece of information can be structurally different.

To understand the ACID vs. BASE characterization, it is actually helpful to define one more acronym, CAP.

  • Consistency - whether system is fully operational according to established rules
  • Availability - whether system available when requested
  • Partition-tolerance - whether system continues to operate even under circumstances of data loss
The CAP theory states that a database system can only have two out of the three. I'm simplifying here, as there are actually many nuances to how these three acronyms define their component words. However, simplistically, SQL systems are generally not partitioned and tend to fulfill consistency over availability by allowing only a master database to be read and edited. And simplistically, a NoSQL system is partition-tolerant and tends to fulfill availability over consistency, by allowing for "best guesses" based on currently available information since the system may be changing continuously.

When would a user prefer either SQL or NoSQL? SQL's consistency and adherence to ACID makes it better for functions like banking and other secured information, because these require precise transactions and accurate answers upon request. SQL is also better for complex queries that consider structured information from many sources. Meanwhile, NoSQL can be useful for web sites that process large numbers of new pieces of information and many queries where a "best guess" answer is adequate. NoSQL may also be a better choice for a business that is growing and changing quickly due to less need for defining structure at initiation.
